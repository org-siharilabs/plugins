# AI Interactions

## What the User Is Doing

The user is working alongside AI — asking their assistant to generate a response, reviewing an AI-suggested reply, watching an automated conversation unfold, or interpreting an AI classification. They are deciding whether to trust, correct, or override what the AI produces. The interface must make this collaboration feel natural, transparent, and safe.

## Emotional Intention

**"I understand what the AI is doing and I'm in control."** Users should never feel like observers to an autonomous process. AI is powerful but it is a tool the user wields — not an agent operating independently. Every AI state, output, and suggestion should be clearly attributed, interpretable, and correctable. Transparency builds trust; opacity erodes it.

Reference: [design-ethos] — "Trustworthy by default. Never fake human."

## How It Works

### AI Processing States

| State | Duration | Visual Pattern | Token |
|-------|---------|---------------|-------|
| Thinking | 0–3s | Teal pulse on AI indicator avatar | `cv-bg-accent-subtle` opacity loop |
| Working | 3–30s | Teal pulse + *"Working…"* text label | `cv-status-info` text + pulse |
| Long-running | 30s+ | Progress bar (if measurable) + elapsed time | `cv-bg-accent` progress fill |
| Streaming output | Active | Cursor blinking at the end of streaming text | Inline cursor, `cv-text-primary` |
| Complete | — | Checkmark icon with brief success fade | `cv-status-success` icon, 1s fade |
| Failed | — | Coral/red icon + error message + solution path | `cv-status-error` + Alert |

The teal pulse is the canonical AI working indicator. It must stop the instant the first token of output begins streaming. Do not continue pulsing during streaming — it is confusing.

### Uncertainty and Language

Chatverce AI must never overstate confidence. The interface communicates certainty in two tiers:

**Confident:** *"This looks like a billing question."* / *"I found 3 matching contacts."*

**Uncertain:** *"I'm not sure — this might be about returns or about shipping."* / *"I couldn't find an exact match."*

Rules:
- Never display raw confidence scores (0.87 confidence) — users cannot act on percentages
- Never present uncertain output as certain — qualify language when the AI is unsure
- AI suggestions are **visually distinct** from user-authored content: use `cv-bg-accent-subtle` background on AI-suggested content, and a small AI attribution label ("Suggested by assistant")

### Correction and Control

The user is always in control. Every AI output or suggestion must support:

- **Edit** — user can modify the AI-generated text before using it
- **Regenerate** — user can request a new version of the same output
- **Reject** — user can dismiss the suggestion entirely without consequence
- **Show reasoning** (where applicable) — a "Why?" link that expands a brief explanation of the AI's decision

Never use AI to take irreversible actions automatically. AI can suggest; the user confirms. The single exception is automated conversation handling (the core product feature) — but even there, users set the rules in advance and can intervene at any time.

### Transparency

- Communicate AI involvement visually. If a response was generated by an AI assistant, show the assistant's avatar and name — never present it as coming from the user's team.
- Never fake human. If a customer asks "Am I talking to a human?", the assistant must be capable of an honest answer. This is enforced in the product, not left to the user's prompt.
- In onboarding and first-encounter tooltip, explain: *"Your assistant handles conversations automatically. You can review, edit, or take over at any time."*
- AI-generated content is attributed. Manually edited content loses the AI attribution label — it is now the user's content.

## Components Used

- **AI working indicator** — teal pulse on assistant avatar or response area
- **Spinner** — only when the AI operation is brief and the content area is small
- **Toast** — AI operation completion (background tasks) or failure
- **Alert** — AI failure requiring user action; persistent inline within the conversation
- **Progress bar** — long-running AI operations (>30s) where duration is estimable
- **Badge** — "AI" attribution badge on assistant-generated content

## Platform Considerations

### Web (Next.js + Tailwind + shadcn/ui)

**Streaming text:** Render AI responses character-by-character (or in token chunks) as they stream from the server. The streaming cursor is a blinking `|` character appended to the current output. Use `cv-duration-fast` blink interval.

**Inline indicators:** The AI working indicator sits inside the conversation thread, directly in the position where the response will appear. It occupies the same vertical space as the eventual response to prevent layout shift.

**Keyboard:** AI suggestion popups must be keyboard-navigable. `Tab` moves to the suggestion actions (Edit, Regenerate, Reject). `Escape` dismisses the suggestion.

### iOS (Expo + NativeWind)

**Haptic on completion:** `Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success)` when a long-running AI operation completes. Not for every response — only for operations over 5 seconds where the user has been waiting.

**Streaming on mobile:** Same streaming character-by-character render. Ensure the `ScrollView` containing the conversation auto-scrolls to follow the streaming output (scroll to end on new character) unless the user has manually scrolled up.

### Android (Expo + NativeWind)

Same streaming and haptics pattern as iOS. On Android, the `expo-haptics` success notification maps to a vibration pattern. Check availability — no-op gracefully on devices without haptics.

## Do / Don't

| Do | Don't |
|----|-------|
| Show the teal pulse while AI is processing; stop it immediately when output begins | Continue pulsing the AI indicator during streaming output |
| Use qualified language for uncertain AI output ("I'm not sure…") | Present uncertain AI output with the same confidence as certain output |
| Visually distinguish AI-suggested content from user-authored content | Mix AI suggestions into the user's workflow without attribution |
| Always provide Edit, Regenerate, and Reject for every AI suggestion | Lock the user into accepting or rejecting AI output with no editing path |
| Attribute AI-generated content with the assistant's name and avatar | Present AI responses as coming from the user's own team |
| Never take irreversible actions automatically without user confirmation | Auto-delete, auto-archive, or auto-reply to all conversations without user-defined rules |
| Explain AI involvement during onboarding | Bury the AI transparency statement in fine print |

## Chatverce-Specific

**Autopilot conversations:** When an AI assistant handles a conversation autonomously, the conversation thread shows the assistant's avatar and name on each AI message. A persistent "Handled by [Assistant Name]" label at the top of the thread makes this unambiguous.

**Human takeover:** The user can take over any AI-handled conversation at any time. A "Take over" button is always visible on AI-handled conversations. When clicked: the assistant is paused for this conversation, the conversation moves to the user's manual queue, and a brief transition label: *"You're now handling this conversation."*

**AI suggestion panel:** When Chatverce suggests a reply template or categorization, it appears as a panel below the message composer — not as text inserted into the composer. The user must explicitly accept (copies to composer) or reject (dismisses panel) the suggestion. Never auto-populate the composer with AI text.

**Confidence in routing:** Chatverce routes conversations to assistants based on message content. When routing confidence is low, show a routing alert: *"This conversation was assigned to [Assistant], but it might be about [topic]. Reassign?"* with a reassign button. This surfaces the uncertainty without blocking the workflow.

**Failed AI response:** If an assistant fails to respond (API error, timeout, content policy), show an inline Alert in the conversation thread: *"[Assistant Name] couldn't respond. Retry or handle manually."* Include a Retry button and a "Handle manually" link that triggers the human takeover flow.
